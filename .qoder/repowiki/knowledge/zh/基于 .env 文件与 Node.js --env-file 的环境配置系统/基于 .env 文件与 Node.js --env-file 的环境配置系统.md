## 配置系统概览

该项目采用**轻量级环境变量驱动**的配置方式，通过 `.env` 文件 + Node.js `--env-file` 参数加载运行时配置，未引入第三方配置库（如 config、nconf 等）。

## 核心机制

### 1. 多环境配置文件
- `server/.env.development` — 开发环境：PostgreSQL 连接信息、`NODE_ENV=development`
- `server/.env.production` — 生产环境：同上，`NODE_ENV=production`
- `server/.env` — 默认兜底配置（本地直接运行时使用）

所有数据库连接参数统一以 `POSTGRES_*` 前缀命名：`POSTGRES_HOST`、`POSTGRES_PORT`、`POSTGRES_DATABASE`、`POSTGRES_USER`、`POSTGRES_PASSWORD`。

### 2. 加载策略
- **Node.js 原生加载**：`package.json` 的 `start`/`dev` 脚本使用 `node --env-file=.env.xxx src/index.js`，由 Node.js 18+ 原生支持，无需额外依赖。
- **dotenv 兼容导入**：`server/src/db.js` 和迁移脚本通过 `import 'dotenv/config'` 作为兼容层，确保在旧版 Node 或特定场景下也能加载 `.env`。

### 3. 配置读取位置
| 模块 | 读取方式 | 关键变量 |
|------|----------|----------|
| `server/src/index.js` | `process.env.PORT` / `process.env.HOST` | 服务端口、监听地址 |
| `server/src/db.js` | `process.env.POSTGRES_*` | PostgreSQL 连接池配置 |
| `server/src/middleware/auth.js` | `process.env.JWT_SECRET` | JWT 签名密钥（带硬编码回退值） |
| `next.config.mjs` | 静态常量 | Next.js rewrites 目标后端地址（硬编码 `localhost:3333`） |

### 4. 前端配置
- 前端无独立配置文件，Next.js 构建时通过 `next.config.mjs` 中的 `rewrites` 将 `/api/*` 和 `/uploads/*` 转发至 `http://localhost:3333`。
- 该转发规则为**硬编码**，未从环境变量读取，前后端耦合于固定端口 3333。

## 架构约定

1. **单一配置源**：所有运行时配置来自 `.env` 文件，不区分 JSON/YAML/TOML 等多格式。
2. **按环境拆分文件**：`.env.development` 与 `.env.production` 分离，通过启动脚本选择。
3. **集中式数据库配置**：仅 `db.js` 一处读取数据库相关环境变量，其他模块通过 `initDb()` 获取连接池。
4. **敏感信息处理**：JWT 密钥存在硬编码回退值（`moke-blog-secret-key-2024`），在生产环境中应严格覆盖。
5. **缺少类型校验**：环境变量直接以字符串形式读取，部分需手动转换（如 `parseInt(process.env.POSTGRES_PORT)`），无 schema 验证。

## 开发者规范

- 新增配置项应在 `server/.env.development` 中定义默认值，并在 `db.js` 或对应模块中读取。
- 禁止在前端代码中直接引用后端敏感配置；如需暴露给前端，应通过 API 返回白名单字段。
- 部署时应通过容器编排或平台环境变量注入覆盖 `.env` 文件内容，避免将真实凭据提交到版本控制。
- 建议后续引入 `zod` 或 `joi` 对 `process.env` 进行解析与校验，提升健壮性。
