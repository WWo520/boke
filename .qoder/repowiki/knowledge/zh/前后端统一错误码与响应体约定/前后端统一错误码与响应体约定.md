本仓库在后端 Express 与前端 Next.js 之间采用统一的「HTTP 状态码 + 结构化 error 对象」错误处理约定，并通过中间件集中完成鉴权类错误拦截。

## 1. 后端（Express）错误模式
- 业务错误：在路由 handler 中直接 res.status(4xx).json({ error: { code, message } }) 返回，如 NOT_FOUND、FORBIDDEN、VALIDATION_ERROR、EMAIL_EXISTS、SLUG_EXISTS、HAS_POSTS、INVALID_CREDENTIALS、INVALID_PASSWORD、FILE_TOO_LARGE、NO_FILE、UPLOAD_ERROR 等。
- 未捕获异常：每个可能抛错的 handler 包裹 try/catch，catch 分支统一 console.error(...) 后返回 500 + { error: { code: 'SERVER_ERROR', message } }。
- 全局错误中间件：未发现 app.use((err, req, res, next) => ...) 形式的 Express 全局错误处理器；异常由各自 try/catch 兜底。
- 上传错误：通过 multer 回调中的 err instanceof multer.MulterError 区分 LIMIT_FILE_SIZE 等子错误，映射为带 code 的 JSON。
- 认证/授权错误：集中在 server/src/middleware/auth.js 的 authMiddleware（401 UNAUTHORIZED/TOKEN_INVALID/USER_NOT_FOUND）和 adminMiddleware（403 FORBIDDEN）。

## 2. 前端（Next.js）错误模式
- 统一请求封装：src/api/client.js 定义 ApiError extends Error，携带 status、code、message；request() 对非 res.ok 响应解析 json.error.code/message 并抛出 ApiError。
- 网络/解析异常：JSON 解析失败抛出 PARSE_ERROR，XMLHttpRequest 层对 error/abort/load 事件分别抛出 NETWORK_ERROR、ABORTED、PARSE_ERROR。
- 页面级 404：src/app/not-found.jsx 提供用户友好的「页面未找到」UI，配合 Next App Router 的默认 404 机制。
- 调用方处理：各 API 模块（auth/posts/columns 等）仅暴露 Promise-returning 函数，上层组件自行 catch ApiError 展示 Toast 或跳转登录。

## 3. 关键文件
- server/src/index.js — 应用入口、multer 配置、内联路由及 try/catch 错误归口
- server/src/middleware/auth.js — JWT 鉴权与管理员校验的错误返回
- server/src/routes/auth.js — 注册/登录/改密等业务错误码示例
- server/src/routes/posts.js — 文章 CRUD 的错误码使用范式
- src/api/client.js — 前端 ApiError 类型与统一 fetch/XMLHttpRequest 错误转换
- src/app/not-found.jsx — 客户端 404 页面

## 4. 开发者应遵循的规则
- 所有业务分支错误必须返回 { error: { code, message } }，禁止裸字符串或无 code 的响应。
- 新增路由需包裹 try/catch，catch 中记录日志并以 500 SERVER_ERROR 兜底。
- 鉴权/权限相关错误一律走 authMiddleware / adminMiddleware，不要在各路由重复实现。
- 前端调用统一通过 api.* 方法，捕获 ApiError 后根据 code 做差异化 UI 反馈（如 401 跳登录、403 提示无权限）。
- 上传接口错误按 multer 子错误分类，映射到语义化 code（如 FILE_TOO_LARGE）。
